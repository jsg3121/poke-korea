# ADR-0002: 기능 개선 로드맵 및 다중 Phase 체이닝 브랜치 전략

- **상태**: 승인
- **날짜**: 2026-04-06
- **담당**: jsg3121, claude

## 맥락

사용자 유입 증대를 위해 국내외 포켓몬 정보 사이트를 조사한 결과, 다수의 도구/계산기/콘텐츠 기능 추가가 필요함이 확인됨. 작업 규모가 크고 API 의존도에 따라 3단계(Phase)로 나뉘어, 기존 단일 버전 브랜치 전략만으로는 관리가 어려움.

핵심 요구사항:
- Phase별로 버전을 나누어 changelog와 작업 범위를 명확히 분리
- main에는 모든 Phase 완료 후 한 번에 배포 (불필요한 중간 배포 방지)
- 필요 시 특정 Phase까지만 먼저 배포할 수 있는 유연성

## 결정

**다중 Phase 체이닝 브랜치 전략**을 도입한다.

### 브랜치 구조

```text
main
 └→ feature/1.37.0 (Phase 1: API 변경 없이 가능한 기능)
      └→ feature/1.38.0 (Phase 2: 클라이언트 계산 도구)
           └→ feature/1.39.0 (Phase 3: API 확장 필요)
```

- 각 Phase 루트는 이전 Phase 루트에서 분기
- 작업 브랜치는 해당 Phase 루트를 향해 PR
- 최종 배포는 마지막 Phase(`feature/1.39.0`) → `main` PR 1회

### 기능 로드맵

| Phase | 버전 | 기능 | API 변경 |
|-------|------|------|:--------:|
| 1 | v1.37.0 | 포켓몬 비교 도구, 소셜 공유 버튼, 즐겨찾기/마이 도감 | 없음 |
| 2 | v1.38.0 | 데미지 계산기, 팀 빌더 | 최소 |
| 3 | v1.39.0 | 성격 도감, 아이템 도감, 진화 조건 상세화 | 필요 |

## 근거

| 기준 | 평가 |
|------|------|
| 기존 워크플로우 호환 | Phase 내부는 기존과 동일 (루트 → 작업 → PR → 머지) |
| changelog 구조 호환 | `changelog/blog/{version}/` 폴더가 Phase별로 자연스럽게 분리 |
| 중간 배포 유연성 | 특정 Phase 루트에서 main PR 가능 (전략 변경 시) |
| Phase간 코드 연속성 | 체이닝이므로 이전 Phase 코드가 자연스럽게 포함 |

## 대안

| 대안 | 장점 | 단점 | 불채택 사유 |
|------|------|------|-------------|
| 단일 버전 루트 (방식 A) | 관리 단순 | Phase 구분 불명확, changelog 혼재 | 8개 이상 기능이 하나의 버전에 섞임 |
| Phase별 독립 + main 각각 머지 | Phase별 독립 배포 | 중간 배포가 필요 없는 상황에서 불필요한 main 머지 다수 | 한 번에 배포하려는 요구사항과 불일치 |

## 결과

- `workflow.md`에 "다중 Phase 체이닝 워크플로우" 섹션 추가
- Phase 1 시작 전 `feature/1.36.0` (현재 하네스 엔지니어링) main 머지 선행 필요
- 이후 main에서 `feature/1.37.0` 분기하여 Phase 1 시작
