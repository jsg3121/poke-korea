# 리서치 스킬

사용자의 의사결정을 지원하기 위해 외부 정보를 조사하고 보고서 형태로 정리한다.

## 사용 판단 기준

이 스킬(서브에이전트)은 별도 컨텍스트 윈도우에서 실행되므로, CLAUDE.md + 하네스 문서 재로딩 + WebFetch 결과 누적으로 토큰 소비가 크다. **작업 규모에 맞는 도구를 선택**해야 한다.

### 메인 대화에서 직접 처리 (이 스킬 사용 X)

- 단순 질문/비교: "A와 B 차이가 뭐야?", "X 라이브러리 장단점 알려줘"
- 1~2개 공식 문서 확인으로 충분한 경우
- 기존 보고서(`research/reports/`)에 이미 조사된 내용인 경우

### 이 스킬 사용 (O)

- **보고서 수준**의 체계적 조사가 필요한 경우
- 3개 이상의 기술/도구를 비교 분석해야 하는 경우
- ADR 작성을 위한 근거 자료 수집이 목적인 경우
- 조사 결과를 `research/reports/`에 보존해야 하는 경우

### 에이전트 사용 (도메인 전문가)

- 조사 + 설계 + 구현이 결합된 복잡한 작업인 경우
- 예: "일본어 랜딩 페이지를 SEO 최적화해서 만들어줘" → seo-specialist 에이전트

> **Why:** 서브에이전트 1회 실행 시 하네스 로딩(~200줄) + 웹 조사(수천~수만 토큰)가 발생한다. 단순 질문에 서브에이전트를 띄우면 동일 정보를 메인 대화 대비 2~3배 토큰으로 처리하게 된다.

## 조사 우선순위

반드시 아래 우선순위를 따른다:

1. **공식 문서** (최우선) — 해당 기술/도구의 공식 사이트, 공식 가이드, 공식 블로그, RFC/스펙
2. **신뢰도 높은 기술 글** — 검증된 기술 블로그(web.dev, MDN, Smashing Magazine 등), 저명한 개발자 블로그, 공신력 있는 벤치마크
3. **커뮤니티 자료** — Stack Overflow, GitHub Discussions, DEV.to 등 (참고 수준)

출처가 불명확하거나 신뢰도가 낮은 자료는 근거로 사용하지 않는다.

## 조사 프로세스

1. 사용자의 질문/요청에서 핵심 조사 주제를 파악한다
2. WebSearch/WebFetch를 활용하여 공식 문서를 우선 탐색한다
3. 공식 문서에서 충분한 정보가 없으면 신뢰도 높은 기술 글을 추가 탐색한다
4. 수집한 정보를 보고서 형태로 정리한다

## 보고서 저장

조사 완료 후 보고서를 `.claude/research/reports/`에 저장한다.

### 파일명 규칙

```
RES-{번호}-{주제}.md
```

- `RES-`: 기술 리서치 보고서 접두사
- `{번호}`: 3자리 순번 (001, 002, ...)
- `{주제}`: kebab-case로 주제 요약 (예: `nextjs-vs-remix`, `auth-library-comparison`)

### 번호 부여

1. `.claude/research/reports/` 폴더에서 기존 `RES-` 파일 확인
2. 가장 높은 번호 + 1로 새 번호 부여
3. 폴더가 없거나 파일이 없으면 001부터 시작

## 보고서 형식

`references/template-res.md` 템플릿을 기반으로 작성한다.

## 주의사항

- 이 스킬은 **코드를 작성하지 않는다**. 조사와 보고만 수행한다.
- 추측이나 의견을 사실처럼 서술하지 않는다. 확인된 사실과 의견을 명확히 구분한다.
- 참고 자료의 URL은 실제로 접근 가능한지 확인한 것만 포함한다.
- 조사 결과가 프로젝트의 기존 ADR과 충돌하는 경우, 해당 ADR을 명시하고 차이점을 설명한다.
