# 코드 리뷰 스킬

PR 생성 전에 변경된 코드를 사전 검토하여 잠재적 이슈를 발견한다. Gemini 코드 리뷰의 보완 역할이다.

## 리뷰 프로세스

### 1. 변경 파일 수집

현재 브랜치에서 대상 브랜치 대비 변경된 파일 목록을 확인한다:

```bash
git diff {대상 브랜치}...HEAD --name-only
git diff {대상 브랜치}...HEAD
```

대상 브랜치를 사용자에게 확인한다. 지정하지 않으면 현재 브랜치의 upstream 또는 `feature/*` 브랜치를 기준으로 한다.

### 2. 서브에이전트 리뷰 실행

**사전 맥락 없이** 서브에이전트를 생성하여 순수하게 코드만으로 리뷰한다. 이는 현재 대화의 맥락에 영향받지 않는 객관적 검토를 위함이다.

서브에이전트에게 전달하는 리뷰 지침:

#### 2-1. 잠재적 이슈 검출

- 런타임 에러 가능성 (null/undefined 접근, 배열 범위 초과 등)
- 로직 오류 (조건 분기 누락, 무한 루프 가능성 등)
- 보안 취약점 (XSS, 인젝션 등)
- 성능 문제 (불필요한 리렌더링, 메모리 누수 가능성 등)
- 엣지 케이스 미처리

#### 2-2. 코드 개선 제안

- 복잡도를 줄일 수 있는 리팩토링 기회
- 중복 코드 제거 가능 여부
- 더 적합한 API/패턴이 있는 경우
- 가독성 개선 제안

#### 2-3. 타입 안전성 검증

- `any` 타입 사용 여부
- `as` 타입 단언 사용 여부 (불가피한 경우 주석 확인)
- `unknown` 대신 `any`를 쓴 경우
- 타입 가드 누락 (타입 좁히기가 필요한 곳)
- `null`/`undefined` 체크 누락
- 제네릭 활용 가능한 곳에서 구체 타입 반복

#### 2-4. 프로젝트 컨벤션 준수

`.claude/conventions/` 가이드를 참조하여 확인:

- 화살표 함수 사용 여부
- const 우선 사용 여부
- JSDoc 함수 설명 여부
- 네이밍 컨벤션 (컴포넌트 PascalCase, 유틸 camelCase)
- import 정렬 순서
- Astro vs Svelte 컴포넌트 사용 기준 (정적 = .astro, 인터랙티브 = .svelte)

### 3. 리뷰 결과 보고

```markdown
## 코드 리뷰 결과

### 요약
- 검토 파일: {N}개
- 🔴 심각 이슈: {N}건
- 🟡 개선 권장: {N}건
- 🔵 참고 사항: {N}건

### 🔴 심각 이슈 (반드시 수정)

#### [{파일경로}:{라인}] {이슈 제목}
{문제 설명}

```{언어}
// 현재 코드
{문제 코드}
```

```{언어}
// 개선 제안
{수정 코드}
```

### 🟡 개선 권장 (수정 권장)

#### [{파일경로}:{라인}] {이슈 제목}
{설명 및 개선안}

### 🔵 참고 사항 (선택적)

- {파일경로}: {참고 내용}

### 타입 안전성

- ✅ `any` 사용 없음 / ❌ `any` 사용 {N}건 발견
- ✅ 불필요한 `as` 없음 / ❌ `as` 단언 {N}건 발견
- {기타 타입 관련 사항}
```

## 주의사항

- 서브에이전트는 **사전 맥락 없이** 코드만으로 리뷰한다. 이는 의도적인 설계이다.
- 리뷰 결과에서 자동 수정은 하지 않는다. 사용자가 검토 후 결정한다.
- 프로젝트 컨벤션(`.claude/conventions/`)과 충돌하는 제안은 컨벤션이 우선한다.
- 이 리뷰는 Gemini 코드 리뷰를 **대체하지 않고 보완**한다.
