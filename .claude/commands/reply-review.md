PR 리뷰 코멘트에 대한 답글을 GitHub에 작성해줘.

## 입력

- PR 번호: $ARGUMENTS (인자가 없으면 사용자에게 PR 번호를 질문)

## 선행 조건

- **반드시 같은 대화에서 `/review-pr {PR번호}`가 먼저 실행되어 있어야 한다.**
- `/review-pr`의 분석 결과(동의/검토 필요/반박 분류)를 기반으로 동작한다.
- `/review-pr`이 실행되지 않았다면, 사용자에게 먼저 `/review-pr`을 실행하도록 안내하고 중단한다.

## 코멘트 작성 규칙

### 동의 항목

- GitHub 코멘트를 남기지 **않는다**.
- 코드 수정 작업만 진행한다 (사용자가 요청한 경우).

### 검토 필요 항목 (일부 동의)

- 사용자에게 각 항목별로 방향을 확인한다.
- 리뷰어의 의견과 **다른 방향**으로 결정된 경우에만 코멘트를 남긴다.
- 리뷰어의 의견대로 진행하기로 한 경우에는 코멘트를 남기지 않는다.

### 반박 항목

- 반박 근거를 포함하여 코멘트를 작성한다.
- 근거는 구체적으로: 공식 문서 기준, 현재 사용 중인 버전, 프로젝트 규칙 등을 명시한다.

## 실행 절차

1. `/review-pr` 분석 결과에서 코멘트 대상(반박 + 검토 필요 중 리뷰어와 다른 방향) 추출
2. PR 인라인 코멘트 조회하여 각 코멘트의 `id` 확인 (`gh api repos/{owner}/{repo}/pulls/{PR번호}/comments`)
3. 이미 답글이 달린 코멘트인지 확인 — 각 코멘트의 `id`로 답글 조회 (`gh api repos/{owner}/{repo}/pulls/{PR번호}/comments` 결과에서 `in_reply_to_id` 필드로 확인)
4. 답글이 없는 코멘트에 대해서만 코멘트 작성
5. 코멘트 작성 전 사용자에게 작성할 내용을 보여주고 확인을 받는다
6. 확인 후 GitHub API로 답글 작성 (`gh api repos/{owner}/{repo}/pulls/{PR번호}/comments -f body="{내용}" -f in_reply_to={코멘트ID}`)

## 코멘트 작성 형식

- 코멘트 본문 **맨 앞에 리뷰어를 멘션**한다. (예: `@gemini-code-assist`, `@reviewer-username`)
- 리뷰어의 GitHub 사용자명은 PR 코멘트 조회 결과의 `user.login` 필드에서 확인한다.

```
@{리뷰어} {반박/대안 설명}

**근거:**
- {구체적 근거 1}
- {구체적 근거 2 (있는 경우)}
```

## 주의사항

- 코멘트는 **정중하고 건설적인 톤**으로 작성한다.
- 감정적 표현이나 비난을 포함하지 않는다.
- 반박하더라도 리뷰어의 의도를 인정하면서 왜 다른 방향이 적절한지 설명한다.
- 이미 답글이 달린 코멘트에는 중복 코멘트를 남기지 않는다.
- `gh` CLI 인증이 안 되어 있으면 사용자에게 `gh auth login`을 안내한다.
