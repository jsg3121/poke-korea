# UX-002: 챔피언스 티어 리스트 페이지 UI/UX 분석

- 조사일: 2026-04-19
- 에이전트: ux-designer
- 분석 대상: `/champions/tier`
- 분석 방법: 코드 분석 + 경쟁 서비스 벤치마킹 (Pikalytics, OP.GG 포켓몬 챔피언스, 포케샘플)

---

## 1. 현재 UI/UX 문제점 분석

### 1-1. 정보 계층 구조 문제

**문제 A: 티어 기준 설명이 헤더 부제에 한 줄로 압축됨**

현재 헤더의 부제 텍스트(`사용률 기반 티어 (S: 30%+, A: 15%+, B: 5%+, C: 1%+, D: 1%-)`)는 `text-sm text-gray-500`으로 표시된다. 티어 리스트를 처음 접하는 사용자는 S/A/B/C/D가 어떤 의미인지, 이 사용률이 어떤 모수를 기준으로 산출된 것인지 충분히 이해하기 어렵다. 특히 "D: 1%-"라는 표기는 "1% 미만"인지 "1%에서 마이너스"인지 직관적으로 파악하기 어렵다.

**문제 B: 포켓몬 이름이 표시되지 않음**

`ChampionsTierGroup` 컴포넌트에서 포켓몬 카드는 이미지(48x48)와 사용률(%)만 노출하고 포켓몬 이름을 숨긴다. 이미지를 기억하지 못하는 사용자나 특정 폼(formName)을 구분해야 하는 사용자는 이름 없이 이미지만으로 판단해야 한다. 특히 formName이 있는 포켓몬(`getDisplayName`에서 `${name} (${formName})` 조합)의 경우 구분 자체가 불가능하다.

**문제 C: 포켓몬 수 정보가 맥락 없이 단독 표시됨**

각 그룹 헤더에 `{pokemons.length}종`이라는 정보가 표시되지만, 전체 포켓몬 대비 몇 종인지, 이전 시즌 대비 변화가 있는지 등의 맥락이 없다. 숫자가 큰 의미를 주지 못한다.

**문제 D: 페이지 전체에 요약 지표가 없음**

티어 리스트 진입 시 "현재 메타에서 어떤 티어가 가장 다양한가", "총 몇 종의 포켓몬이 메타에 포함되어 있는가" 같은 요약 정보가 없다. Pikalytics는 상위 12종을 즉각적으로 부각하고, OP.GG는 전체 픽률/밴률 분포를 상단에 배치하는 방식을 쓴다.

### 1-2. 시각적 표현의 한계

**문제 E: 포켓몬 카드 크기(48x48)가 목록 페이지와 차별화되지 않음**

티어 리스트는 메타 분석 도구로서 핵심 페이지임에도, 포켓몬 카드 이미지가 도감 목록(`ChampionsPokemonCard`)과 동일하거나 더 작은 48x48이다. 시각적 위계가 성립되지 않는다.

**문제 F: 티어 배지 색상 체계에 직관성 부족**

현재 배지 색상 체계를 보면 S는 amber(금색), A는 slate(은색), B는 amber-dark(동색)로 설정되어 있어 S/A/B가 금/은/동 메달 체계를 따르는 것으로 보인다. 그러나 C는 emerald(초록), D는 gray로 S→D로 갈수록 강도가 낮아지는 일관된 색상 그라디언트가 아니다. C의 emerald 색상이 B의 amber-dark보다 시각적으로 더 밝고 강렬하게 느껴질 수 있어 C가 B보다 높은 티어로 오인될 가능성이 있다.

**문제 G: 사용률 수치만으로는 차이를 파악하기 어려움**

사용률 텍스트(`{pokemon.usageRate?.toFixed(1)}%`)만 표시되기 때문에, 같은 A티어 내에서 15.1%와 29.9%의 포켓몬이 나란히 놓여도 시각적으로 동등하게 표시된다. 티어 내에서도 상위/하위 격차를 인지할 수 없다.

**문제 H: flex-wrap 레이아웃에서 마지막 행 정렬이 왼쪽 정렬**

`flex flex-wrap gap-3` 구조에서 마지막 행의 카드가 홀수로 남을 경우 왼쪽 정렬로 남는다. 정렬 방식에 일관성이 없으며, 특히 C/D 티어처럼 포켓몬이 적을 때 공간 낭비가 발생한다.

### 1-3. 사용자 흐름 개선점

**문제 I: 티어별 빠른 이동 수단이 없음**

S/A/B/C/D 그룹이 세로 나열될 때, D 티어를 보려면 전체 페이지를 스크롤해야 한다. S 티어에 포켓몬이 많을 경우 D 티어까지 도달하는 스크롤 깊이가 매우 깊어진다. 앵커 내비게이션이나 사이드 목차가 없다.

**문제 J: 포켓몬 클릭 시 어떤 정보로 이동하는지 사전 안내가 없음**

각 포켓몬 카드는 Link로 감싸져 `/champions/list/[pokemonId]`로 이동하지만, hover 상태에서 포켓몬 이름이나 "상세 보기" 텍스트가 없어 클릭 목적지를 예상하기 어렵다. 이름이 없는 상태에서는 특히 더 그렇다.

**문제 K: 필터/정렬 기능 부재**

사용자가 특정 타입의 포켓몬만 필터링하거나, 사용률 역순으로 정렬하거나, 특정 포켓몬을 검색하는 기능이 없다. 도감 페이지(`ChampionsTypeFilter`)에는 타입 필터가 존재하지만 티어 리스트 페이지에는 없다.

**문제 L: 데스크톱과 모바일 컨테이너가 완전히 동일한 구조**

`ChampionsTier.container.tsx` 데스크톱/모바일 버전이 header margin(`mb-8` vs `mb-6`), 섹션 spacing(`space-y-8` vs `space-y-6`), 타이포그래피 크기(`text-2xl` vs `text-xl`, `text-sm` vs `text-xs`) 외에 레이아웃 구조 자체는 동일하다. 데스크톱의 넓은 화면 공간을 활용하는 별도 레이아웃 전략이 없다.

## 2. 경쟁 서비스 벤치마킹

### 2-1. Pikalytics (pikalytics.com/champions)

포켓몬 챔피언스 전용 메타 분석 서비스로 가장 직접적인 경쟁 서비스다.

**레이아웃 구조:**

- 명시적 티어 구분 대신 순위 기반(#1~#12) 연속 나열 방식을 채택한다.
- 상위 12종을 "주요 메타 정의 포켓몬"으로 시각적으로 강조하고, 이후 "차순위 선택지(Next Clicks After The Top 12)"를 별도 섹션으로 분리한다.
- 각 카드에 순위 번호, 포켓몬 스프라이트, 이름, 사용률 퍼센트, 역할 설명 1줄, 메타게임 해석 2줄을 포함한다.

**시사점:** 단순 티어 배지보다 순위 번호와 역할 설명이 메타 이해에 더 직접적으로 기여한다. 사용률 퍼센트만으로는 포켓몬의 역할을 이해하기 어려운 신규 사용자를 위한 문자 기반 해설 접근이 차별점이다.

### 2-2. OP.GG 포켓몬 챔피언스 (op.gg/pokemon-champions)

게임 메타 분석 분야 레퍼런스 서비스다.

**레이아웃 구조:**

- 도감(258종 전체), 기술(919개), 아이템(138개), 타입 차트를 탑 내비게이션으로 제공하여 티어 리스트가 단독 페이지가 아닌 메타 분석 도구 생태계의 일부로 배치된다.
- 챔피언 호버 시 상세 정보가 팝업으로 표시되어 페이지 이탈 없이 미리 볼 수 있다.
- 픽률/밴률/승률을 함께 표시하여 단일 지표(사용률)만으로는 파악하기 어려운 메타 영향력을 다각도로 제시한다.

**시사점:** 호버 프리뷰 패턴은 데스크톱에서 페이지 이탈을 줄이는 효과적인 인터랙션이다. 다중 지표 표시로 메타 분석 깊이를 높인다.

### 2-3. 포케샘플 (pokesample.com/ko/statistics)

국내 포켓몬 메타 분석 서비스로 동일한 한국어 사용자를 타깃으로 한다.

**레이아웃 구조:**

- 표(테이블) 형식으로 순위, 순위 변동, 포켓몬명, 타입을 컬럼으로 구성한다.
- 싱글/더블 배틀 형식을 탭으로 전환할 수 있어 배틀 형식별 비교가 가능하다.
- 순위 변동(이전 주 대비)을 표시하여 메타 트렌드를 즉각 파악할 수 있다.
- 항목 클릭 시 상세 페이지로 이동하는 단순한 플로우를 유지한다.

**시사점:** 테이블 레이아웃은 많은 포켓몬을 밀도 높게 표시하는 데 유리하다. 순위 변동 표시가 메타 트렌드 파악에 결정적 차별점이 된다. 한국어 사용자 친화적 레이블이 진입 장벽을 낮춘다.

### 2-4. u.gg (u.gg/lol/tier-list) — 장르 유사 레퍼런스

포켓몬 전용 서비스는 아니지만 티어 리스트 UI의 업계 표준을 제시하는 서비스다.

**레이아웃 구조:**

- 필터 바(랭크, 지역, 게임 타입, 패치)를 상단에 배치하여 사용자가 맥락을 설정한 뒤 티어 리스트를 열람하도록 유도한다.
- 티어별 섹션 내에서 픽률, 승률, 밴률을 컬럼으로 제공한다.
- "show all champions" 옵션으로 전체 포켓몬을 한눈에 보는 뷰를 제공한다.

**시사점:** 필터 먼저 설정 → 결과 열람 순서의 정보 흐름이 사용자 의도를 명확히 반영한다.

## 3. 개선 방향 제안

### 3-1. 레이아웃 개선안

**요약 헤더 섹션 도입**

현재 제목 + 한 줄 설명에서 벗어나, 총 포켓몬 수/티어별 분포를 수평 요약 카드로 제공한다.

```
[포켓몬 챔피언스 티어 리스트]
사용률 기반 · 총 {n}종 포켓몬 포함

[S] 4종  [A] 8종  [B] 15종  [C] 22종  [D] 9종
```

각 티어 카운트를 클릭하면 해당 섹션으로 앵커 이동한다. 이 요약 바는 사용자가 어느 티어에 먼저 관심을 가질지 결정하는 데 필요한 전체 지도(overview) 역할을 한다.

**데스크톱 2열 레이아웃 도입**

현재 데스크톱이 모바일과 동일한 단일 컬럼 구조다. 데스크톱에서는 좌측 고정 사이드바에 티어 앵커 내비게이션을 배치하고 우측에 티어 그룹 본문을 배치하는 구조가 적합하다.

```
[좌측 사이드 — 고정]       [우측 본문]
  S 티어 (4종)     →        [S 그룹 섹션]
  A 티어 (8종)     →        [A 그룹 섹션]
  B 티어 (15종)    →        [B 그룹 섹션]
  C 티어 (22종)    →        [C 그룹 섹션]
  D 티어 (9종)     →        [D 그룹 섹션]
```

**모바일 앵커 바 도입**

모바일에서는 좌측 사이드바 대신 S/A/B/C/D 탭을 수평 스크롤 가능한 고정 바로 제공한다. 탭 클릭 시 해당 섹션으로 스크롤 이동한다.

### 3-2. 포켓몬 카드 개선안

**이름 표시 추가**

현재 이미지 + 사용률만 표시하는 구조에서 이름을 추가한다. 모바일에서는 카드 너비 제한이 있으므로 이미지 아래에 `text-xs` 2줄 이내로 포켓몬 이름을 표시한다. formName이 있는 경우 괄호 없이 formName만 sub-label로 별도 표시한다.

```
[이미지 64x64]
  리자몽
  (메가 X)    ← formName 별도 표시
  43.2%
```

**이미지 크기 확대**

현재 48x48에서 64x64로 확대한다. 48x48은 작은 아이콘 수준으로, 티어 리스트처럼 포켓몬 간 비교가 핵심 목적인 페이지에서는 인식성이 부족하다. 64x64는 터치 타깃 최소 크기(44px, Apple HIG / 48px, Google Material)를 충족하면서도 상세 이미지 확인이 가능한 크기다.

**hover 상태 개선 (데스크톱)**

현재 `hover:bg-gray-50` 배경색 변경만 적용된다. 개선안은 다음과 같다.

- hover 시 포켓몬 이름 툴팁 또는 카드 내 이름 레이블 노출
- 테두리 강조 (`ring-2 ring-primary-1`)로 선택 상태를 명확히 표시
- 미세한 상향 이동 (`-translate-y-1`) 애니메이션으로 클릭 가능 요소임을 인지

### 3-3. 티어 배지 색상 체계 개선

현재 색상 체계에서 C 티어의 emerald가 B 티어의 amber-dark보다 시각적으로 더 밝게 느껴지는 문제가 있다. S→D로 갈수록 강도가 일관되게 감소하는 색상 시스템을 제안한다.

| 티어 | 현재           | 개선안          | 이유                                                           |
| ---- | -------------- | --------------- | -------------------------------------------------------------- |
| S    | amber(금)      | amber/gold 유지 | 최상위 티어 표현으로 적합                                      |
| A    | slate(은)      | violet/purple   | 은색이 "차갑고 낮은" 인상을 줄 수 있음. 보라색이 강함의 연장선 |
| B    | amber-dark(동) | blue/indigo     | B는 준수한 티어이므로 차가운 계열 중 강한 색상                 |
| C    | emerald(초록)  | teal/cyan       | 중간 강도의 차가운 계열 색상으로 B보다 낮게 인지               |
| D    | gray           | gray            | 낮은 등급, 탈색 표현으로 적합. 유지                            |

단, 기존 배지를 어떤 컨셉으로 설계했는지 팀 내에서 명확히 합의된 경우(예: 실제 포켓몬 게임 랭킹 색상 체계 반영) 변경 전 재논의가 필요하다.

### 3-4. 사용률 시각화 개선

**티어 내 상대 비율 바 도입**

각 포켓몬 카드 하단에 사용률 퍼센트를 텍스트로 표시하는 것에 더해, 같은 티어 내 최고 사용률을 100%로 기준 잡은 미니 바를 추가한다.

```
[이미지]
 이름
 [███░░] 43.2%  ← 티어 내 상대 비율 바
```

바 너비는 Tailwind `width` 인라인 스타일로 구현 가능하며, Chart.js 의존성 없이 처리할 수 있다. 이 방식은 같은 A 티어 내에서도 상위와 하위 포켓몬의 실질적 격차를 한눈에 파악하게 해준다.

### 3-5. 필터/정렬 기능 제안

도감 페이지에 이미 `ChampionsTypeFilter` 컴포넌트가 존재한다. 티어 리스트 페이지에 이를 확장 적용할 수 있다.

- **타입 필터**: 특정 타입 포켓몬만 표시. 모든 티어 섹션에 필터가 동시에 적용됨.
- **이름 검색**: 입력창에 포켓몬 이름을 입력하면 해당 포켓몬이 포함된 티어 섹션만 표시.
- **티어 선택 필터**: S/A/B/C/D 체크박스로 원하는 티어만 표시.

단, 필터를 추가할 경우 현재 SSR 기반 페이지(`revalidate = 86400`)에서 클라이언트 사이드 필터링으로 처리해야 한다. 데이터 로드는 서버에서, 필터 적용은 클라이언트 상태로 분리하는 방식이 적합하다.

### 3-6. 반응형 전략

| 브레이크포인트              | 레이아웃 전략                                                    |
| --------------------------- | ---------------------------------------------------------------- |
| 모바일 (max-width: 768px)   | 단일 컬럼, 상단 S/A/B/C/D 앵커 탭 바 고정, 카드 64x64, 이름 표시 |
| 데스크톱 (min-width: 769px) | 좌측 앵커 사이드바(고정) + 우측 티어 그룹 본문, 카드 80x80       |

**모바일 특화:**

- 앵커 탭 바를 `sticky top-[header-height]` 고정으로 배치하여 스크롤 중에도 티어 이동 가능
- 각 티어 섹션에 `id="tier-s"` 등 앵커 ID 부여

**데스크톱 특화:**

- 사이드바 너비 `w-40`으로 좁게 유지하여 본문 공간 확보
- 스크롤 위치에 따라 활성 티어 사이드바 항목을 하이라이트하는 스크롤스파이 패턴 적용

### 3-7. 인터랙션 패턴

**로딩 상태:**

- 티어 그룹별 스켈레톤 UI: 배지 영역 + 카드 영역을 블록 스켈레톤으로 표시
- 현재 SSR 방식이므로 클라이언트 로딩이 발생하지 않지만, 필터 기능 추가 시 필터 전환 중 스켈레톤 적용이 필요

**빈 상태 (포켓몬이 없는 티어):**

- 현재 `pokemons.length === 0`이면 `null`을 반환하여 해당 티어 섹션 자체가 사라진다. 이는 사용자가 "D 티어가 없는 건지, 렌더링 오류인지" 구분하지 못하게 한다.
- 개선안: 빈 티어도 섹션을 유지하되 "현재 시즌 데이터가 없습니다" 메시지를 표시

**포켓몬 카드 클릭 피드백:**

- 현재 `transition-colors`만 적용. `transition-all duration-150`으로 확장하여 scale + 배경색 변화를 함께 적용한다.

## 5. 핵심 요약

### 즉시 적용 가능한 개선 (Low 난이도, High 임팩트)

1. 포켓몬 이름 표시 추가 — 이미지 아래에 `text-xs` 레이블 추가
2. 이미지 크기 48x48 → 64x64 확대 — `w-12 h-12` → `w-16 h-16` 변경

### 중기 개선 (스프린트 단위 권장)

1. 상단 앵커 탭 바 (S/A/B/C/D) — 스크롤 깊이를 근본적으로 해결
2. 티어 내 상대 비율 미니 바 — Tailwind width 인라인 스타일로 Chart.js 없이 구현 가능
3. 데스크톱 2열 레이아웃 — 컨테이너 분리 구조 활용

### 가장 큰 경쟁 우위 차별점

포케샘플이 "순위 변동(트렌드)" 정보를 제공하는 반면, 현재 포케코리아 티어 리스트는 정적 스냅샷만 제공한다. 이전 시즌 대비 티어 변화 정보(S→A로 하락, 신규 진입 등)를 추가하면 메타 추적 도구로서의 재방문 동기를 만들 수 있다. 이는 Pikalytics가 순위 기반 내러티브로 차별화한 방향과도 맥락이 같다.
