# 1.45.0 — 리서치 보고서 정정

> **작업 일자**: 2026-05-28
> **작업 브랜치**: `feature/1.45.0` (직접 작업)

## 📋 작업 개요

**작업 유형**: 문서 정정 + 에이전트 지침 보강
**담당**: jsg3121 + Claude

2026-05-26 작성된 MI-BA 리서치 보고서와 ADR-0005에서 사실 오류가 확인되어 폐기 처리하고, 동일한 실수가 재발하지 않도록 MI/BA 에이전트 정의에 가드 지침을 추가했습니다.

## 🔍 발견된 오류

BA 분석이 `.claude/specs/metrics-baseline.md`의 "페이지별 트래픽 TOP 10"만 보고 다음 두 페이지를 **"구현 안 됨, 신규 생성 필요"** 로 단정했으나, 실제 상용 코드를 확인한 결과 두 페이지 모두 이미 구현되어 있었습니다.

| 페이지          | BA 보고서 주장                       | 실제 상태                                                              |
| --------------- | ------------------------------------ | ---------------------------------------------------------------------- |
| `/moves/[id]`   | "구현 안 됨, 신규 생성 필요 (1순위)" | **구현됨** — 메타데이터, JSON-LD, 반응형 desktop/mobile, ISR 1년 완비  |
| `/ability/[id]` | "구현 안 됨, 신규 생성 필요 (3순위)" | **구현됨** — 동일 패턴                                                 |

**원인**: BA 에이전트가 `src/app/` 라우트 구조를 직접 확인하지 않고 통계 문서만 보고 추정한 것. 통계는 "트래픽 분포"를 보여줄 뿐 "구현 유무"를 보여주지 않습니다.

## 🛡 재발 방지 지침 요약

MI/BA 에이전트는 이제 자사 콘텐츠/기능 점유 영역을 언급하기 전 다음을 **순서대로 반드시 수행**해야 합니다.

1. `src/app/` 라우트 구조 직접 확인
2. 동적 라우트(`[id]`, `[slug]`)의 `page.tsx` 직접 읽기
3. 메타데이터/SEO 자산 적용 여부 확인
4. 통계 문서와 코드 상태의 정합성 확인

또한 BA의 갭 분석에서 다음 두 케이스를 명시적으로 구분해야 합니다.

| 케이스                                  | 후속 작업 방향                                  |
| --------------------------------------- | ----------------------------------------------- |
| A. 페이지/기능 자체가 부재              | 신규 구현                                       |
| B. 구현은 되어 있으나 트래픽이 적음     | 콘텐츠 보강 / SEO 메타 개선 / 내부 링크 / 색인 |

두 케이스는 후속 작업이 완전히 다르므로 혼동 시 잘못된 우선순위가 도출됩니다.

## 🔜 후속 작업

폐기된 분석을 대체할 새 BA 재분석이 필요합니다. 이미 구현된 페이지의 트래픽이 낮은 진짜 원인(콘텐츠/SEO/색인)을 다시 진단한 후 새 ADR과 함께 작성될 예정입니다.
